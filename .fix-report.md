# Watcher-sync review fixes (SoT-042/043)

Branch tip at start: `c8cc597` (feat/watcher-sync, detached HEAD). All work committed on the same detached HEAD.

## Critical 1 — spawn-ordering race (watcher exits at birth)

**(a) Reorder the spawn.** `src/services/hook-service.ts`: moved the
`spawnDetachedWatcher(...)` call out of the top of `handleSessionStart` (it
used to run before `startSession`/`resumeSession` write the cwd session
pointer) to immediately after the `sessionId`/`additionalContext` resolution
block, i.e. after both the resume path (`resumeSession`) and the new-session
path (`startSession`) have written the pointer the watcher's liveness check
reads.

**(b) Startup grace, defense in depth.** `src/services/watcher-service.ts`:
`watcherShouldExit` now accepts an optional `{ startedAtMs, graceMs }`. When
the cwd has **zero** session pointers (the exact race window — the pointer
literally hasn't been written yet) and we're still within `graceMs` of
`startedAtMs`, it returns `false` (not-exit) instead of `true`. Existing
pointers (even stale ones) are judged normally — only the "nothing here yet"
case gets the grace. `runWatcherLoop` captures `loopStartedAt = Date.now()`
and passes `{ startedAtMs: loopStartedAt, graceMs: pollMs }` on every
`shouldExit` call. Deps-injected `shouldExit` mocks in tests are 0/2-arg
functions and ignore the extra arg, so no existing test needed behavior
changes here.

**RED/GREEN:** `tests/unit/watcher-service.test.ts` — new test *"grace: does
not exit before the first poll interval when no session pointer exists yet"*
uses the **real** default `watcherShouldExit` (not a mock) with an empty
sandbox (no pointer file anywhere) and asserts the first tick still runs.
Verified RED against the pre-fix source (reverted via `git stash`): the old
code has no `WatcherShouldExitOpts`/lease params at all, so the test suite
fails to run cleanly against it (see "RED verification" below). GREEN:
`ticks: 1, exit: 'max-ticks'`.

Not independently testable at the hook-service level: `handleSessionStart`
has no injectable spawn seam and the suite forces
`MEMORIZE_WATCHER_DISABLED=1` globally (see `vitest.config.ts`), so ordering
is proven by code inspection + the loop-level grace test covering the
fallback path.

## Critical 2 — per-tick `sync.state.updated` churn + dead idle gate

**(a) No-op pull writes zero events.** `src/services/sync-service.ts`
`pullProject`: removed the unconditional `syncStatus: 'syncing'` flip before
the network round trip, and made the trailing `syncStatus: 'idle'` write
conditional (`if (state.syncStatus !== 'idle')`) — mirrors `pushProject`'s
existing empty-payload guard. A no-op pull (nothing new, already idle) now
appends **zero** `sync.state.updated` events. Real pulls that insert events
are unaffected: `applyPullResponse` still writes the watermark
(`lastPulledEventId`) unconditionally when `response.lastRemoteEventId` is
present.

**(b) Gate excludes bookkeeping events.** `src/services/watcher-service.ts`
`watcherTick`: `hasSelfLaneEvents` now also requires
`event.type !== 'sync.state.updated'` — the same filter `buildPushPayload`
already applies to the actual push payload. Without this, a real pull's own
watermark-bookkeeping event (self-lane by construction — `sourceProjectId`
defaults to the local `projectId`) always looked like local work, so the
gate never gated a foreign-only pull.

**RED/GREEN:**
- `tests/integration/sync-service.test.ts` — new test *"pullProject appends
  zero events when the remote has nothing new"*. RED (via `git stash` on
  `src/services/sync-service.ts` only): `expected 5 to be 3` (2 extra churn
  events). GREEN after restoring the fix.
- `tests/integration/auto-sync.test.ts` — extended the existing "no-op
  autoPull ... leaves the pull watermark unchanged" test with an event-count
  assertion (same RED/GREEN shape). Note: an `.some(type === 'sync.state.updated')`
  absence-assertion I initially added was **wrong** (the log legitimately
  already contains one from the setup's real clone-pull) and I removed it —
  the length-equality check is the correct "no NEW event" assertion; recorded
  here per the review contract's "say so" instruction.
- `tests/unit/watcher-service.test.ts` — new focused test *"gate skips the
  push call when the only delta past watermark is sync.state.updated
  bookkeeping"* (uses the new `WatcherTickDeps.push` injection seam to count
  calls, not just the final `pushed` count — see note below on why the count
  alone doesn't distinguish pre/post-fix) plus a real end-to-end test (see
  Minor section) that exercises this exact scenario over a real file
  transport with a genuinely foreign `sourceProjectId`.

**Design note:** `pushProject`'s own empty-payload short-circuit (pre-existing)
already made `watcherTick`'s final `pushed` count 0 in both the pre- and
post-2b-fix code for a pure-bookkeeping delta — the bug was *wasted work*
(a full local scan + `autoPush` call), not a wrong final count. To make this
testable I added an optional `WatcherTickDeps` seam to `watcherTick` (mirrors
`WatcherLoopDeps`) so a test can assert `push` was never **called**, not just
that its reported count was zero.

## Important A — per-tick lock self-verification + PID-reuse lease

`src/services/watcher-service.ts`:
- `WatcherLockInfo` gained `renewedAt?: string`.
- New `isLeaseStale(holder)`: stale when `renewedAt` (falling back to
  `startedAt`) is older than `2 * pollMsFromEnv()`, **regardless of pid
  liveness** — this is what reclaims a crashed watcher on a Windows host
  where the pid has since been reused.
- `acquireWatcherLock`'s takeover condition is now
  `!holder || isLeaseStale(holder) || !isPidAlive(holder.pid)` (was pid-only).
- New `renewWatcherLock(projectId, pid)`: re-reads the lock, returns `false`
  if the pid no longer matches ours (self-verification), otherwise rewrites
  the file with a fresh `renewedAt` (the lease refresh).
- `runWatcherLoop` calls `renewLock` (defaults to `renewWatcherLock`) once
  per iteration, before the tick body; a `false` result exits immediately
  with `exit: 'lock-lost'` (new `WatcherLoopResult` variant) — converts a
  double-acquire into self-healing within one tick, as specified.

**Test:** `tests/unit/watcher-service.test.ts` — *"exits lock-lost when the
lock is stolen by a different pid mid-loop"* (foreign pid written into the
lock file inside a mocked `tick`, loop exits `lock-lost` on the next
iteration) and *"treats a lease older than 2x the poll interval as stale even
with a live pid"* (holder pid is the test's own — genuinely alive — process,
only the expired lease permits takeover). Plus two smaller unit tests for
`renewWatcherLock` itself (refreshes `renewedAt`; reports `false` once
another pid holds the lock). RED confirmed: running the new tests against
the pre-fix source crashed the vitest worker (old `acquireWatcherLock`/lock
shape doesn't support any of this).

## Important B — a throwing tick kills the daemon

`runWatcherLoop`: the tick call is now wrapped in `try/catch` inside the
loop. On throw: warn to stderr with a `N/5` counter, `sleep(pollMs)`,
`continue` (does not advance `ticks`, matching existing "ticks = successful
ticks" semantics). A success resets the counter to 0. After 5 **consecutive**
failures, returns `{ ticks, exit: 'tick-failures' }`.

**Test:** *"contains a throwing tick: N-1 failures then a success keep the
loop alive"* (3 throws then success, loop reaches `max-ticks` normally) and
*"exits tick-failures after 5 CONSECUTIVE throws"*.

## Minors

- **Atomic marker write.** `writeInboundMarker` now uses `writeJson` from
  `storage/fs-utils.ts` (backed by `write-file-atomic`) instead of a raw
  `writeFile`.
- **Configured predicate.** `watcherTick` now also requires
  `state.remoteProjectId`, aligning with `auto-sync-service.ts`'s
  `isConfigured`. New test: a project with `syncEnabled` + `syncTransport`
  but no `remoteProjectId` now reports `configured: false` instead of
  polling forever. No `WatcherLoopResult` change needed — `'not-configured'`
  already existed.
- **Real-transport end-to-end test.** New `describe('watcherTick — real
  transport, end-to-end (SoT-043)')` in `tests/unit/watcher-service.test.ts`,
  using the real `createFileSyncTransport` (no mocking of pull/push): seeds a
  remote with a genuinely foreign event (`sourceProjectId` set to a different
  store id, simulating a shared-memory union-lane contributor) → tick pulls
  it, writes the marker, and — thanks to 2a+2b — does **not** push it back;
  empty remote → tick → no new marker; a real local task → tick → pushes,
  remote gains it; another foreign-only event → tick → pulls it but does not
  push (gate holds). This one test pins the full SoT-043 semantics this
  review fixed.

## Concerns / follow-ups

- The hook-service spawn-ordering fix (Critical-1a) has no direct unit test
  (no seam, and the suite disables the watcher globally). Coverage is
  code-review-level plus the loop's independent startup-grace test.
- `runWatcherLoop`'s lease renewal and tick-failure containment both run
  every iteration; on a very short test-only `pollMs` this adds one extra
  small file write per tick in production — negligible at the real 30s
  cadence.
- Did not touch `watcher.ts` CLI (`src/cli/commands/watcher.ts`) — its
  single outer `acquireWatcherLock`/`releaseWatcherLock` pair around
  `runWatcherLoop` needed no change; the loop now self-manages the lease
  internally using the same lock file.
